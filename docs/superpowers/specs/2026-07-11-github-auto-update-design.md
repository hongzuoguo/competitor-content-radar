# GitHub Releases 自动更新与开源发布设计

## 目标

将“对标内容雷达”发布为公开开源项目，并让 Windows 桌面应用在启动后自动检查、下载和安装新版本。当前 0.1.1 用户只需再手动安装一次包含更新器的版本，之后升级均可在应用内完成。

## 公开仓库

- 仓库：`hongzuoguo/competitor-content-radar`
- 可见性：公开
- 许可证：MIT License
- 默认分支：`main`
- GitHub Releases 同时作为用户下载入口和自动更新源
- 源代码、构建脚本和公开安装包不得包含 API Key、GitHub Token、抖音会话、飞书凭证、本地数据库或用户媒体文件

## 方案选择

采用 `electron-updater` 配合 GitHub Releases。相比自建更新服务器，该方案不增加服务器运维；相比自定义下载器，它复用 Electron 生态的版本比较、增量文件、安装器衔接和更新事件。

暂不支持 Microsoft Store、强制更新、静默安装或多更新通道。首个版本仅提供稳定通道。

## 更新流程

1. 仅在已打包的生产应用中启用更新器；开发环境不请求 GitHub。
2. 应用启动并显示主窗口后检查一次更新。
3. 没有新版本时不弹窗，仅记录“已是最新版本”。
4. 发现新版本后自动后台下载，并通过受控 IPC 将状态和百分比发送到渲染进程。
5. 下载完成后，如果没有采集或分析任务正在运行，应用自动正常停止调度器、托盘和数据库连接，再调用更新器退出并安装。
6. 如果任务正在运行，更新器等待任务结束，再自动重启安装；不会中断作品下载、转写、AI 分析或数据写入。
7. 顶部栏显示“更新已下载，将在当前任务完成后自动安装”或“正在自动更新”，但不要求用户点击确认按钮。
8. 自动更新失败时保留当前可用版本，并在之后启动或用户手动重试时再次检查。

更新不会删除 Electron `userData` 目录，因此博主列表、分析记录、模型文件、API Key 和本地设置均保留。

## 模块边界

### 主进程更新服务

新增独立的 `UpdateService`，只负责：

- 配置 GitHub 更新源
- 触发检查和下载
- 将 `idle`、`checking`、`available`、`downloading`、`ready`、`up_to_date`、`error` 状态标准化
- 在应用空闲时安装已下载版本
- 把底层错误转换为不泄露路径或凭证的用户消息

主进程入口负责服务生命周期；更新服务不访问产品数据库。

### IPC

预加载层只暴露以下最小能力：

- 读取当前更新状态
- 订阅更新状态变化并返回取消订阅函数
- 手动重新检查

渲染进程不能传入任意下载地址、文件路径或命令。

### 界面

顶部栏展示紧凑状态：

- 检查中：不阻塞操作
- 下载中：展示百分比
- 已就绪且任务运行中：展示“任务完成后自动更新”
- 已就绪且应用空闲：展示“正在自动更新”
- 失败：展示“重试”，应用其余功能保持可用
- 最新版：默认不长期占据界面

更新状态使用 `aria-live` 宣读，按钮具备明确名称和键盘焦点。

## 发布配置

Electron Builder 配置 GitHub provider：

- owner：`hongzuoguo`
- repo：`competitor-content-radar`
- channel：stable
- Windows target：NSIS x64

每个 Release 必须上传：

- Windows 安装程序 `.exe`
- 安装程序 `.blockmap`
- `latest.yml`

版本号使用语义化版本。Git 标签与 Release 名称使用 `v<version>`，例如 `v0.2.0`。发布凭证只存在于发布环境，不写入仓库或桌面应用。

## 开源清理

首次推送前执行：

- 检查 Git 历史和当前文件是否包含密钥、Token、Cookie、个人数据库或媒体文件
- 保留 `.env`、`release/`、`node_modules/`、日志、用户数据目录的忽略规则
- 添加 MIT `LICENSE`
- 添加 README，说明功能、安装、首次模型下载、抖音风险验证、AI 费用和飞书当前状态
- 明确本项目仅处理用户有权访问的公开内容，使用者应遵守平台规则和相关法律

## 错误与安全

- 无网络、GitHub 不可达或限流：进入可重试错误状态，不影响采集和分析
- 下载或校验失败：不执行安装，保留当前可用版本
- 未下载完成或业务任务仍在运行时禁止调用安装
- 更新事件移除监听器，避免窗口重建后重复通知
- 公共 GitHub Release 不需要在客户端嵌入 GitHub Token
- Windows 代码签名不属于本次范围；未签名安装包可能触发 SmartScreen 提醒，后续可增加证书签名

## 测试与验收

自动化测试覆盖：

- 更新器事件到公共状态的转换
- 下载进度范围处理
- 未就绪时拒绝安装
- IPC 参数与事件订阅边界
- 顶部栏在下载、等待任务、自动安装和失败状态下的文案
- 更新下载完成后仅在业务任务空闲时触发安装
- 开发环境禁用远程更新检查

打包验收覆盖：

1. 旧版本安装并保留测试数据。
2. GitHub 发布更高版本。
3. 旧版本发现并下载新版本。
4. 重启安装后版本号更新。
5. 原有博主、设置和分析数据仍存在。
6. 断网时应用可正常启动和使用。

## 非目标

- 不在业务任务运行期间强行中断更新
- 不要求用户手动确认或重新下载安装包
- 不实现 nightly、beta 等多通道
- 不在客户端保存 GitHub 发布 Token
- 不在本次工作中接入商业代码签名证书
